# Tally — Payments & Billing API

A Stripe-style payments API built with **FastAPI + SQLModel**. Tally is
look-alike server #1 in the [MCPCloud showcase](./example-servers.md) — a real,
deployed, callable API used to demonstrate ingesting an OpenAPI spec, enriching
terse tool descriptions, generating a typed MCP server, and running it through
the platform end to end.

It is intentionally **terse**: operation summaries are auto-generated by FastAPI
from function names (`Confirm Payment Intent`, `Read Mrr`), there are no
descriptions and no examples. That "before" state is the setup for MCPCloud's
enrichment payoff — see [example-servers.md §5](./example-servers.md).

## Quick start

```bash
uv run fastapi dev
```

That installs dependencies into a managed environment and starts the server on
`http://127.0.0.1:8000` with auto-reload. It creates a local SQLite database and
seeds the demo tenant automatically on first run. Use `uv run fastapi run` for a
production-style start (no reload).

- OpenAPI spec (the ingestion URL): `http://127.0.0.1:8000/openapi.json`
- Interactive docs: `http://127.0.0.1:8000/docs`
- Health check: `http://127.0.0.1:8000/healthz`

## Authentication

Every endpoint requires an API key in the `Authorization` header:

```bash
curl -H "Authorization: Bearer tally_sk_demo_4f9a2b7c1e8d6053" \
  http://127.0.0.1:8000/reports/mrr
```

The key maps to an isolated demo tenant; all data is scoped to it. Override it
with the `DEMO_API_KEY` environment variable.

## Data model

`Customer`, `PaymentMethod`, `PaymentIntent`, `Charge`, `Refund`, `Plan`,
`Subscription`, `Invoice`. Amounts are integer minor units (cents).

A `PaymentIntent` moves through `requires_payment_method` →
`requires_confirmation` → `succeeded` / `requires_action` / `canceled`.
Confirming an intent of **$500 or more** returns `requires_action` instead of
succeeding (so the flow is demoable); smaller intents succeed and create a
`Charge`.

## Endpoints (21)

`customers` (4), `payment-methods` (2), `payment-intents` (5), `refunds` (1),
`charges` (1), `plans` (1), `subscriptions` (3), `invoices` (2),
`reports` (revenue + MRR). Full list in
[example-servers.md §5.4](./example-servers.md). `GET /reports/mrr` normalises
active subscriptions to a monthly figure (annual plans ÷ 12) and excludes trials
and past-due subscriptions.

## Seed data

`uv run python -m app.seed` wipes and re-seeds the demo tenant (this is what the
nightly reset job runs). The seed is deterministic: 25 customers, 6 plans, 15
subscriptions in mixed states, 19 invoices (several deliberately `open` /
overdue), and payment intents stuck at `requires_payment_method` — so the
companion "Failed Payment Recovery" skill has real work to find.

## Drift-detection demo (v1.1)

The v1.1 `/disputes` endpoints (`GET /disputes`,
`POST /disputes/{id}/respond`) ship in the codebase but are **hidden by
default**. Set `ENABLE_DISPUTES=true` and restart to expose them — re-fetching
`/openapi.json` then surfaces the two new operations, which is the drift demo
in [example-servers.md §5.6](./example-servers.md).

## Configuration

| Variable | Default | Purpose |
|---|---|---|
| `DATABASE_URL` | `sqlite:///./tally.db` | SQLAlchemy URL; Fly.io sets Postgres automatically |
| `DEMO_API_KEY` | `tally_sk_demo_4f9a2b7c1e8d6053` | Accepted bearer key |
| `SEED_ON_STARTUP` | `true` | Seed the demo tenant when the database is empty |
| `ENABLE_DISPUTES` | `false` | Expose the v1.1 `/disputes` endpoints |

## Deployment (Fly.io)

Live at **https://api.tally.mcpcloud-demo.com** — app `tally-mcpcloud-demo`
in the `mcpcloud-demos` org, backed by Fly Postgres, behind a Let's Encrypt
cert (DNS managed in Cloudflare, "DNS only" / unproxied).

To reproduce the provisioning from scratch:

```bash
fly apps create tally-mcpcloud-demo --org mcpcloud-demos
fly postgres create --name tally-mcpcloud-demo-db --org mcpcloud-demos
fly postgres attach tally-mcpcloud-demo-db --app tally-mcpcloud-demo  # sets DATABASE_URL
fly secrets set DEMO_API_KEY=tally_sk_demo_4f9a2b7c1e8d6053 --app tally-mcpcloud-demo
fly deploy
fly certs add api.tally.mcpcloud-demo.com --app tally-mcpcloud-demo   # then add A/AAAA in DNS
```

### Nightly reset

A scheduled Fly Machine (`tally-nightly-reset`, process group `reset`) runs
`python -m app.seed` once every 24h to wipe and re-seed the demo tenant. It is
separate from the `app` process group, so `fly deploy` leaves it untouched.

```bash
fly machine run \
  --app tally-mcpcloud-demo --schedule daily --restart no \
  --name tally-nightly-reset --region iad --vm-memory 512 \
  --metadata fly_process_group=reset \
  registry.fly.io/tally-mcpcloud-demo:<current-deployment-tag> \
  -- python -m app.seed
```

The reset Machine is **pinned to the image tag it was created with** — a later
`fly deploy` does not update it. After a deploy that changes the data models or
the seed script, recreate it: `fly machine destroy tally-nightly-reset --force`,
then re-run the command above with the new deployment tag (`fly image show`).
