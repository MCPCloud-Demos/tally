## Quick start

```bash
uv run fastapi dev
```

That installs dependencies into a managed environment and starts the server on
`http://127.0.0.1:8000` with auto-reload. It creates a local SQLite database and
seeds the demo tenant automatically on first run. Use `uv run fastapi run` for a
production-style start (no reload).

- OpenAPI spec (the ingestion URL): `http://127.0.0.1:8000/openapi.json`
- Interactive docs: `http://127.0.0.1:8000/docs`
- Health check: `http://127.0.0.1:8000/healthz`

## Authentication

Every endpoint requires an API key in the `Authorization` header:

```bash
curl -H "Authorization: Bearer tally_sk_demo_4f9a2b7c1e8d6053" \
  http://127.0.0.1:8000/reports/mrr
```

The key maps to an isolated demo tenant; all data is scoped to it. Override it
with the `DEMO_API_KEY` environment variable.

## Data model

`Customer`, `PaymentMethod`, `PaymentIntent`, `Charge`, `Refund`, `Plan`,
`Subscription`, `Invoice`. Amounts are integer minor units (cents).

A `PaymentIntent` moves through `requires_payment_method` â†’
`requires_confirmation` â†’ `succeeded` / `requires_action` / `canceled`.
Confirming an intent of **$500 or more** returns `requires_action` instead of
succeeding (so the flow is demoable); smaller intents succeed and create a
`Charge`.

## Seed data

`uv run python -m app.seed` wipes and re-seeds the demo tenant (this is what the
nightly reset job runs). The seed is deterministic: 25 customers, 6 plans, 15
subscriptions in mixed states, 19 invoices (several deliberately `open` /
overdue), and payment intents stuck at `requires_payment_method` â€” so the
companion "Failed Payment Recovery" skill has real work to find.

## Drift-detection demo (v1.1)

The v1.1 `/disputes` endpoints (`GET /disputes`,
`POST /disputes/{id}/respond`) ship in the codebase but are **hidden by
default**. Set `ENABLE_DISPUTES=true` and restart to expose them â€” re-fetching
`/openapi.json` then surfaces the two new operations, which is the drift demo

## Configuration

| Variable | Default | Purpose |
|---|---|---|
| `DATABASE_URL` | `sqlite:///./tally.db` | SQLAlchemy URL; Fly.io sets Postgres automatically |
| `DEMO_API_KEY` | `tally_sk_demo_4f9a2b7c1e8d6053` | Accepted bearer key |
| `SEED_ON_STARTUP` | `true` | Seed the demo tenant when the database is empty |
| `ENABLE_DISPUTES` | `false` | Expose the v1.1 `/disputes` endpoints |

## Deployment (Fly.io)

```bash
fly launch --no-deploy
fly postgres create --name tally-db
fly postgres attach tally-db          # sets DATABASE_URL
fly secrets set DEMO_API_KEY=tally_sk_demo_4f9a2b7c1e8d6053
fly deploy
```

Target host per the plan: `https://api.tally.mcpcloud-demo.com`.
